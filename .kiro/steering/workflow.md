# 작업 방식

## 기본 원칙
- 기능 구현 요청 시 먼저 tasks.md를 작성하여 작업 계획을 세운다
- tasks.md를 사용자에게 확인을 받고 Phase별로 코드를 구현한다
- git 명령어(브랜치 생성, 커밋, push)는 사용자가 직접 수행한다
- Kiro는 tasks.md + 코드 작성 + 테스트 JSON + PR 본문을 제공한다

## 작업 순서
1. tasks.md 작성 (Phase별 작업 계획 + 브랜치/커밋 정보 포함)
2. 사용자 확인 후 Phase별 코드 구현 (파일 생성/수정)
3. 컴파일 오류 확인
4. API Dog용 테스트 JSON 제공
5. PR 본문 제공 (아래 템플릿 사용)
6. git 명령어 안내 (브랜치명, 커밋 메시지)

## PR 본문 템플릿
```
## 기능 설명
{한 줄 요약}

## 작업 상세 내용
- [x] {구현한 파일/기능 목록}

## 확인한 내용
- [ ] 로컬 실행 확인
- [ ] 주요 기능 동작 확인
- [ ] 에러 로그 확인
- [ ] 기존 기능 영향 여부 확인

## 스크린샷 / 실행 결과

## 기타 공유사항

## 관련 티켓
- closes #SOFIT-{번호}
```

## 커밋 컨벤션
- 형식: `[SOFIT-{번호}] {태그}: {작업 내용}`
- 브랜치: `feat/SOFIT-{번호}-{기능명-kebab-case}`
- PR 베이스: `dev`

## 주의사항
- git 명령어를 직접 실행하지 않는다 (사용자에게 안내만)
- 사업자등록번호는 하이픈 없는 10자리
- 전화번호는 하이픈 없는 11자리
- PIN은 6자리
- 응답은 한국어로
