# 작업 방식

## 기본 원칙
- 기능 구현 요청 시 **충분한 대화**를 통해 요구사항을 명확히 한 뒤 tasks.md를 작성한다
- tasks.md 확인 후 커밋 단위별로 코드를 구현한다
- **커밋 단위마다 멈추고** git 명령어를 안내한다
- **Phase(PR 단위) 마지막 커밋**에서는 테스트 케이스 + PR 정보를 함께 제공한다
- git 명령어(브랜치 생성, 커밋, push)는 **사용자가 직접 수행** — Kiro는 절대 실행하지 않는다

## 작업 흐름

### 1단계: 요구사항 파악 (대화)
- 사용자의 요청을 듣고 모르는 부분은 질문한다
- API 명세, ERD, 비즈니스 로직, 예외 처리 등을 확인한다
- 충분히 파악될 때까지 대화를 이어간다

### 2단계: tasks.md 작성
- Phase(PR 단위)별 작업 계획 수립
- Phase 안에 커밋 단위를 명시
- 사용자 확인 후 구현 시작

### 3단계: 커밋 단위 구현 루프
1. 코드 구현 (파일 생성/수정)
2. 컴파일 확인 (`gradlew compileJava`)
3. **멈추고** 커밋 안내:

```
---
✅ 커밋 완료 시점

💻 git 명령어:
git add {파일 목록}
git commit -m "{커밋 메시지}"

다음 커밋 진행하려면 "다음" 해주세요.
---
```

### 4단계: Phase 마지막 커밋 (PR 올리기)
Phase의 마지막 커밋일 때는 테스트 + PR 정보를 함께 제공:

```
---
✅ Phase {N} 완료 — PR 올릴 시점입니다.

🧪 테스트 케이스:
| 케이스 | 요청 | 기대 결과 |
|--------|------|-----------|
| ... | ... | ... |

📋 PR 제목: {제목}

📋 PR 본문:
{본문 전체}

💻 git 명령어:
git add {파일 목록}
git commit -m "{마지막 커밋 메시지}"
git push -u origin {브랜치명}
---
```

## 커밋 컨벤션
- 형식: `[SOFIT-{번호}] {태그}: {작업 내용}`
- 브랜치: `feat/SOFIT-{번호}-{기능명-kebab-case}`
- PR 베이스: `dev`

## PR 본문 템플릿
```
## 기능 설명
{한 줄 요약}

## 작업 상세 내용
- [x] {구현한 파일/기능 목록}

## 확인한 내용
- [ ] {구체적 테스트 케이스 1}
- [ ] {구체적 테스트 케이스 2}
- [ ] ...

## 스크린샷 / 실행 결과

## 기타 공유사항

## 관련 티켓
- closes #SOFIT-{번호}
```

## 주의사항
- git 명령어를 직접 실행하지 않는다 (안내만 한다)
- 커밋 단위가 되면 반드시 멈추고 사용자에게 알린다
- 사용자가 "다음" 또는 "진행"이라고 하면 다음 커밋을 시작한다
- 테스트 불가능한 커밋(설정, DTO 등)은 커밋 명령어만 안내한다
- 테스트 가능한 시점(Controller 완성 등)에서만 테스트 케이스를 제공한다
- "확인한 내용"에는 실제 API Dog 등으로 테스트 가능한 케이스를 구체적으로 적는다
- 응답은 한국어로
