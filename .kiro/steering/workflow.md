# 작업 방식

## 기본 원칙
- 기능 구현 요청 시 먼저 tasks.md를 작성하여 작업 계획을 세운다
- tasks.md 확인 후 Phase별로 코드를 구현한다
- **커밋 단위(Phase 완료)마다 멈추고** 사용자에게 git 작업을 안내한다
- git 명령어(브랜치 생성, 커밋, push)는 **사용자가 직접 수행**한다 — Kiro는 절대 실행하지 않는다

## 작업 흐름
1. tasks.md 작성 → 사용자 확인
2. Phase별 코드 구현 (파일 생성/수정)
3. 컴파일 확인 (`gradlew compileJava`)
4. **멈추고** 커밋 안내 (브랜치명 + 커밋 메시지 제공)
5. 사용자가 git 작업 완료 후 다음 Phase 진행
6. 모든 Phase 완료 후: API Dog 테스트 JSON + PR 본문 제공

## 커밋 시점 안내 형식
```
---
✅ Phase {N} 완료 — 커밋 시점입니다.

브랜치: feat/SOFIT-{번호}-{기능명}
커밋 메시지: [SOFIT-{번호}] Feat: {작업 내용}

커밋 후 다음 Phase 진행할게요.
---
```

## 커밋 컨벤션
- 형식: `[SOFIT-{번호}] {태그}: {작업 내용}`
- 브랜치: `feat/SOFIT-{번호}-{기능명-kebab-case}`
- PR 베이스: `dev`

## PR 본문 템플릿
```
## 기능 설명
{한 줄 요약}

## 작업 상세 내용
- [x] {구현한 파일/기능 목록}

## 확인한 내용
- [ ] 로컬 실행 확인
- [ ] 주요 기능 동작 확인
- [ ] 에러 로그 확인
- [ ] 기존 기능 영향 여부 확인

## 스크린샷 / 실행 결과

## 기타 공유사항

## 관련 티켓
- closes #SOFIT-{번호}
```

## 주의사항
- git 명령어를 직접 실행하지 않는다 (안내만 한다)
- 커밋 단위가 되면 반드시 멈추고 사용자에게 알린다
- 사용자가 "다음" 또는 "진행"이라고 하면 다음 Phase를 시작한다
- 응답은 한국어로
